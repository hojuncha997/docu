HTTP(Hyper Text Transfer Protoco)

개략
HTTP는 분산되고 협응하는 하이퍼미디어 정보시스템을 위한 어플리케이션 계층의 프로토콜이다.

기술적 개략

HTTP는 클라이언트-서버 컴퓨팅 모델의 request-respond 프로토콜로서 기능한다. 예를 들어
웹브라우져가 클라이언트이고, 웹사이트를 호스팅하는 컴퓨터에서 구동중인 어플리케이션이 서버이다.
클라이언트는 서버에게 HTTP request 메시지를 보낸다. HTML 파일 등과 같은 리소스를 제공하는, 또는
클라이언트의 행위에 따라 그 외의 기능을 할 수도 있는, 서버는 클라이언트에게 response 메시지를 보낸다.

이 response는 요청에 대한 완료 정보일수도 있고, 메시지의 바디에 요청된 컨텐츠를 담고 있을 수도 있다.

web browser는 대표적인 user agent(UA)이다.

HTTP는 클라이언트와 서버 사이의 중간단계의 네트워크 요소들을 향상시키고 소통을 가능케 하기 위해 디자인 되었다.
트래픽이 높은 웹사이트들은 리스폰스 타임을 향상시키기 위해 업스트림 서버 대신 사용하는 웹캐쉬 서버로부터 때때로
이득을 얻었다. 웹 브라우져는 네트워크 트래픽을 줄이기 위해 이전에 접근했던 웹 리소스를 캐슁했다가 다시 사용했다.
사설 네트워크 범위의 HTTP proxy server는 외부의 서버를 릴레잉하여 메시지를 전달함으로써 globally routabl
address 없이도 클라이언트들의 커뮤니케이션을 가능케 했다.

HTTP는 internet protoco suit의 프레임워크 내에서 디자인 된 어플리케이션 레이어 프로토콜이다. 그것의
정의는  기본으로 깔려 있고 신뢰할 만한 트랜스포트 레이어 프로토콜 위에서 추정된다.
TCP(Transmission control Protocol)이 흔히 사용된다. 그러나 HTTP는 UDP(user datagram protocol),
처럼 신뢰하기 어려운 프로토콜에서도 사용될 수 있다. HTTPU나 SSDP가 그 예다.

HTTP 리소스는 네트워크 상엣 URL(Uniform Resource Locators)에 의해 식별되고 놓여진다. 이러한
URL은 HTML문서에서 하이퍼링크로 인코딩 된다. 

---

HTTP의 세션

HTTP 세션은 네트워크 상에서의 request-response 트랜젝션의 연속적인 과정을 의미한다.
HTTP 클라이언트는 서버의 특정 포트로 향하는 TCP 연결을 생성함으로서 리퀘스트를 시작한다.
서버는 그 포트를 listening 하면서 클라이언트로의 request 메시지를 기다린다. 리퀘스트가 도착하면,
서버는 클라이언트에게 status line을 회신한다. 예를 들면 아래와 같은 것이다.

"HTTP/1.1 200 OK" 그리고 요청된 메시지도 보낸다.

메시지의 바디에는 요청된 리소스가 들어있다. 그러나 error나 다른 정보가 담겨 반환될 수도 있다.
